# 优化指南

[English](optimizing-guide.md)

面向用户的 bundle-plugin 及单个技能优化指南。涵盖快速开始、范围检测、「诊断 → 委派 → 验证」流程、8 个优化目标、A/B 评估和反馈迭代。

## 概述

优化（optimizing）在 hub-and-spoke 架构中是迭代改进的**编排器（orchestrator）**。它**诊断**需要改什么，将实质性内容编辑**委派**给 `bundles-forge:authoring`，并通过调用 `bundles-forge:auditing`**验证**结果（单向：审计只出报告，不会回调优化）。

与审计（仅评估）不同，优化从枢纽驱动改进 — 修复描述、减少 token、收紧工作流链路、处理用户反馈 — 并在适当时将 SKILL.md 与 agent 内容工作路由到 authoring。

**核心原则：** 为 Agent 的体验而优化。每项改进都应该让技能更容易被发现、更快加载、更清晰可循。

> **权威来源：** 完整的执行协议（范围检测、目标详情、A/B 评估步骤、反馈验证）位于 `skills/optimizing/SKILL.md`。本指南帮助你理解*每个目标做什么*、*如何解读结果*、*何时使用哪种方法*。

---

## 快速开始

改进现有 bundle-plugin 或技能的最快方式：

1. **运行审计**（可选但推荐）— `bundles-forge:auditing` 生成优化可直接消费的诊断报告
2. **调用** `/bundles-optimize`（或请 agent 优化你的项目或技能）
3. **Agent 自动检测范围** — 项目根目录或单个技能 — 并选择适用的目标
4. **审查提议的变更** — agent 在应用前呈现其改进方案
5. **验证** — agent 在变更后运行 `bundles-forge:auditing` 确认改进效果

优化技能接受本地路径、GitHub URL 和 zip/tar.gz 文件作为输入。你不需要先克隆仓库。

---

## 选择路径

优化会从你提供的路径自动检测范围：

| 范围 | 检测方式 | 模式 | 适用目标 |
|------|---------|------|---------|
| 项目根目录 | 包含 `skills/` + `package.json` | **项目优化** | 全部 8 个目标 + 反馈 |
| 单个技能目录 | 包含 `SKILL.md`，无 `skills/` 子目录 | **技能优化** | 核心目标（1-4）+ 反馈 |

### 决策流程图

```
你要优化整个项目还是单个技能？
  ├─ 整个项目 → 项目优化（全部 8 个目标）
  │    ├─ 有审计报告？ → 作为输入提供，按优先级优化
  │    └─ 没有审计报告？ → 优化技能自行诊断
  └─ 单个技能 → 技能优化（目标 1-4 + 反馈）
       ├─ 技能触发了但结果不对？ → 反馈迭代
       └─ 技能需要工程改进？ → 目标 1-4
```

### 支持的输入

| 输入 | 操作 |
|------|------|
| 本地目录路径 | 直接使用 |
| GitHub 仓库 URL | 浅克隆到临时目录 |
| GitHub 子目录 URL | 克隆仓库，提取子目录 |
| Zip/tar.gz 文件路径 | 解压到临时目录 |

如果下载失败，技能会报告错误并建议替代方案（提供本地路径或 zip 文件）。

### 输入来源

优化可以消费先前审计的报告。常见模式是**先审计、再优化** — 但请记住 **`bundles-forge:auditing` 仅做诊断**：它不会调用优化。**你**在需要修复时运行优化，或由其他编排器（例如 **`bundles-forge:releasing`**）在其流水线中编排审计与优化的顺序。

| 输入 | 来源 | 用途 |
|------|------|------|
| `audit-report` | `bundles-forge:auditing`（完整项目） | 所有 8 个目标的逐技能分解 |
| `skill-report` | `bundles-forge:auditing`（技能模式） | 技能优化的聚焦 4 类报告 |
| `workflow-report` | `bundles-forge:auditing`（工作流模式） | 目标 4 的 W1-W9 发现 |
| `user-feedback` | 直接来自用户 | 迭代过程的行为反馈 |

## 流程：诊断 → 委派 → 验证

优化对审计是**单向**验证关系，不是技能间相互耦合的循环。审计从不调用优化；优化在变更后需要验证报告时会**调用**审计。

```
optimizing 诊断 → 将内容编辑委派给 authoring → 通过 auditing 验证
```

**重要：** 若验证仍有问题，交给用户手动决定 — 不要无限循环。后续轮次是用户或编排器**重新**运行优化（或 authoring），而不是审计「回调」优化。

### 推荐工作流

1. 可选：运行 `bundles-forge:auditing`（完整、技能或工作流模式），生成作为输入的诊断报告。
2. 审查报告 — 优先处理严重发现。
3. 以审计报告（或你的目标）运行 `bundles-forge:optimizing`。它会诊断、按需将内容工作委派给 `bundles-forge:authoring`、按其协议处理非内容类优化，并**调用** `bundles-forge:auditing` 做变更后验证（由**优化**触发该步骤 — 审计不会自动启动优化）。
4. 审查验证报告 — 若仍有问题，手动决定是否再跑一轮优化或 authoring。

## 核心目标（1-4）

这些目标同时适用于项目优化和单技能优化。

### 目标 1：技能描述触发准确性

**影响最大的优化。** 描述是技能发现的主要机制 — 当用户说话时，Agent 将意图与描述字段匹配。

**关键规则：** 描述必须陈述*触发条件*，而非*工作流摘要*。

| | 差（工作流摘要） | 好（触发条件） |
|---|---|---|
| **模式** | "Use for auditing — scans structure, checks manifests, scores categories, generates report" | "Use when reviewing a bundle-plugin for structural issues, version drift, or before release" |
| **问题** | Agent 走描述捷径而不读完整 SKILL.md | Agent 读取完整技能获取执行细节 |
| **结果** | 跳过步骤、不完整执行 | 按设计完整执行 |

**其他规则：**
- 必须以 "Use when..." 开头
- 保持在 250 字符以内（超过会在技能列表中截断）
- 包含具体的症状、场景和上下文
- 不要提及技能的内部步骤

#### 如何验证描述质量

运行 linter 捕获机械性问题：

```bash
python skills/auditing/scripts/audit_skill.py <project-root>
```

描述相关检查为 **Q3-Q7**：缺失描述（Q3）、"Use when..." 前缀（Q5）、工作流摘要反模式（Q6）、长度 >250 字符（Q7）。完整 lint 套件覆盖 Q1-Q15 和 X1-X3 — 详见快速参考。

对于*行为*质量（正确的提示是否触发正确的技能），使用 A/B 评估。

### 目标 2：Token 效率

频繁加载的技能中每个 token 都会占用每次会话的上下文预算。这对引导技能（每次会话加载）和常触发的技能影响最大。

**目标：**
- SKILL.md 正文 < 500 行
- 引导技能（`using-*`）< 200 行
- 将重量级参考资料移至 `references/`

**技巧：**

| 技巧 | 示例 |
|------|------|
| 交叉引用代替重复 | `See bundles-forge:authoring` 代替复制规则 |
| 一个优秀示例胜过三个平庸示例 | 删除教授相同概念的冗余示例 |
| 将标志文档指向 --help | 引用 `python skills/auditing/scripts/audit_skill.py --help` 代替列出所有标志 |
| 消除项目内冗余 | 不要重复其他技能 `references/` 中已有的内容 |

### 目标 3：渐进式加载

三级加载系统确保最小上下文占用：

| 级别 | 何时加载 | 预算 |
|------|---------|------|
| 元数据（name + description） | 始终在上下文中 | ~100 词 |
| SKILL.md 正文 | 技能触发时 | < 500 行 |
| 参考文件（`references/`） | 按需 | 无限制 |

**何时提取到 references/：**
- SKILL.md 接近 500 行
- 仅在执行时需要的表格或检查清单（不是理解技能用途所需的）
- Agent 逐字复制的模板内容

### 目标 4：工作流链路完整性

消费工作流审计发现以识别和修复工作流问题。工作流审计分两层：

- **脚本自动化（W1-W9）：** 静态图分析和语义检查 — 通过 `python skills/auditing/scripts/audit_workflow.py` 运行
- **仅 Evaluator（W10-W11）：** 链式评估和行为验证 — 需要 `evaluator` agent 派遣

如果没有工作流报告：

```bash
python skills/auditing/scripts/audit_workflow.py <project-root>
python skills/auditing/scripts/audit_workflow.py --focus-skills skill-a,skill-b <root>
```

**修复优先级指南：**

| 发现 | 含义 | 修复方式 |
|------|------|---------|
| W1（未声明循环） | 两个技能互相调用但循环未声明 | 如果有意则在 `## Integration` 中添加 `<!-- cycle:a,b -->`，否则重构 |
| W2（不可达技能） | 技能存在但没有链路指向它 | 添加到引导路由，或声明 `Called by: user directly` |
| W3/W4（缺失 I/O） | 终端技能没有 `## Outputs`，或被引用的技能没有 `## Inputs` | 添加带制品 ID 的章节 |
| W5（制品 ID 不匹配） | 上游 `## Outputs` 和下游 `## Inputs` 使用不同名称 | 对齐反引号包裹的制品 ID |
| W9（占位章节） | Inputs/Outputs 存在但为空或通用 | 编写有意义的语义描述 |
| W10（不对称集成） | 技能 A 说它调用 B，但 B 没说被 A 调用 | 添加缺失的 `**Called by:**` 声明 |

单技能模式下，仅 W9（占位章节）和 W10（不对称集成）适用 — 其余需要项目级图分析。

## 项目专属目标（5-8）

这些目标在单技能模式下跳过。它们需要项目级上下文。

### 目标 5：平台覆盖

识别项目尚未支持的平台。添加新平台请调用 `bundles-forge:scaffolding` — 优化技能本身不生成平台适配器。

### 目标 6：安全修复

修复 `bundles-forge:auditing` 类别 10 的安全发现。常见修复：

| 发现 | 修复 |
|------|------|
| Hook 脚本发起网络调用 | 移除或添加注释说明理由 |
| OpenCode 插件能力过大 | 限定到声明的需求范围 |
| Agent 提示词缺少范围约束 | 添加显式边界 |
| SKILL.md 包含编码/混淆内容 | 剥离或替换为明文 |

### 目标 7：技能与工作流重构

项目的结构性变更：添加技能、替换技能、重组工作流链路、或将技能转换为子代理。此功能原属 blueprinting（场景 D），但因其操作现有项目且不产出设计文档，更适合放在 optimizing 中。

#### 何时使用

| 用户说 | 操作 |
|--------|------|
| "给我的项目添加一个新技能" | 目标 7a — 添加技能，接入工作流 |
| "用更好的技能替换这个" | 目标 7b — 替换并更新引用 |
| "工作流链路需要重组" | 目标 7c — 重构执行路径 |
| "这个技能应该改成子代理" | 目标 7d — 转换为只读代理 |
| "我的项目需要更好的 XX 能力" | 反馈流程 → 可能引导到目标 7a |

#### 添加技能（7a）

最常见的重构操作。流程：

1. 读取现有项目 — 映射技能、工作流图、引导路由
2. 清点新技能 — 来源、结构、frontmatter 质量
3. 对照现有项目检查兼容性（命名、职责、规范）
4. 第三方技能 — 遵循共享集成参考文档（`references/third-party-integration.md`），涵盖许可证、安全审计和集成意图
5. 设计插入点 — 新技能在哪里连接？
6. 应用 — 复制、适配、更新 Integration 声明
7. 验证 — 使用 `--focus-skills` 运行聚焦式工作流审计

#### 替换技能（7b）

与添加相同的兼容性分析，外加：
- 映射旧技能的所有引用
- 更新交叉引用、Integration 声明和路由表
- 工作流审计验证

#### 重组工作流（7c）

当执行链路存在低效：
- 映射当前图谱，识别瓶颈或不必要的交接
- 提出新链路方案（呈现给用户）
- 更新 Integration 声明和路由
- 链式 A/B 评估验证

#### 技能转子代理（7d）

转换候选标准：
- 执行过程产生大量临时上下文（搜索结果、文件内容、日志），且后续步骤不需要这些上下文
- 仅做检查/验证而不修改文件的技能
- 产生结构化报告（自包含输出）的技能
- 可与其他工作并行运行的技能（可选加分项）

转换将执行协议提取到 `agents/<role>.md`，并提供子代理不可用时的回退逻辑。转换后，使用 `evaluator` agent 和测试提示词确认新 agent 正确执行原技能的职责，然后运行 `bundles-forge:auditing` 验证调度/回退逻辑。

### 目标 8：可选组件管理

根据项目演进需求添加、调整或迁移可选插件组件。此目标弥补初始脚手架与项目成熟后所需组件之间的差距。

#### 何时使用

| 信号 | 组件 | 操作 |
|------|------|------|
| 技能将 API 密钥/端点硬编码为 `${VAR}` 环境变量 | `userConfig` | 迁移到 `userConfig` 实现自动用户提示 |
| 审计发现 MCP 服务器没有 `userConfig` 支持的认证 | `userConfig` | 添加 `sensitive: true` 的 `userConfig` 字段 |
| 技能引用外部 SaaS API 但无集成 | `.mcp.json` 或 `bin/` | 添加 MCP 服务器或 CLI — 参考决策树 |
| 技能涉及语言相关的代码智能 | `.lsp.json` | 添加 LSP 服务器配置 |
| 用户需要自定义输出格式 | `output-styles/` | 添加输出风格定义 |
| 插件 MCP 服务器有 npm 依赖 | `${CLAUDE_PLUGIN_DATA}` | 添加 SessionStart 依赖安装钩子 |
| 插件使用 `../` 路径或写入 `${CLAUDE_PLUGIN_ROOT}` | 路径迁移 | 改为使用相对 `./` 路径和 `${CLAUDE_PLUGIN_DATA}` |

#### 工作原理

1. **诊断** — 从审计报告、用户反馈或直接检查中识别信号
2. **决策** — 参考 `skills/scaffolding/references/external-integration.md` 的完整决策树（CLI vs MCP、userConfig 模式、PLUGIN_DATA 模式、LSP 字段、output-styles 格式）
3. **执行** — 调用 `bundles-forge:scaffolding` 的 "添加可选组件" 流程
4. **验证** — 运行 `bundles-forge:auditing` 确认结构完整性和安全合规（特别是新 MCP 服务器和 userConfig 敏感值）

## A/B 评估

A/B 评估是描述变更和反馈驱动改进的核心质量保障机制。它并排比较原始版与优化版。

### 工作原理

```
1. 将技能复制到工作版本（<skill-name>-optimized/）
2. 仅对副本应用更改（不要先覆盖原始版）
3. 创建 5+ 个应该触发此技能的真实测试提示词
4. 并行派遣两个 evaluator agent：
   - Evaluator A："original" 标签 → 用原始技能测试
   - Evaluator B："optimized" 标签 → 用优化技能测试
5. 比较结果 → 呈现给用户
6. 用户决定：采用优化版或丢弃
```

### 比较指标

| 指标 | 含义 |
|------|------|
| **触发率** | 多少提示词正确激活了技能？ |
| **假阴性** | 优化版是否遗漏了原始版捕获的场景？ |
| **假阳性** | 任一版本是否在不相关的提示词上误触发？ |
| **步骤准确性** | Agent 是否遵循了所有步骤，还是走了捷径？ |

### 何时跳过 A/B

| 场景 | 跳过？ | 理由 |
|------|:---:|------|
| 纯添加性变更（新触发短语，无修改） | 是 | 简单验证即可 |
| 结构修复（缺失章节、断裂引用） | 是 | 不是行为变更 |
| 修改现有触发器的描述重写 | **否** | 必须验证无回归 |
| 反馈驱动的行为变更 | **否** | 必须比较新旧行为 |

### 链式 A/B 评估

用于工作流过渡（不是单个描述），使用链式评估：

1. 定义一个真实的端到端场景
2. 以 "chain" 标签和有序技能列表派遣 evaluator
3. 审查每个交接点的过渡质量评级
4. 关注 "broken" 交接 — 表示缺失制品或不清晰的指令

**何时使用链式评估：** 修改 Inputs/Outputs 后、向链路添加技能后、或工作流审计发现指示问题时。

### 子代理回退

当子代理派遣不可用时，两个选项：

| 回退方式 | 方法 | 权衡 |
|---------|------|------|
| 顺序内联 | 按 `agents/evaluator.md` 协议内联执行，随机化顺序 | 较慢，可能有顺序偏差 |
| 跳过 A/B | 直接应用变更并简单验证 | 较快，无对比数据 |

由用户选择使用哪种回退。

## 反馈迭代

当用户报告技能触发了但产出错误结果时，反馈流程提供结构化的迭代。

### 反馈分类

| 用户说 | 操作 |
|--------|------|
| "这个技能触发了但结果不对" | 反馈迭代 |
| "步骤顺序不对" | 反馈迭代 |
| "描述格式不符合规范" | 优化目标 1-3 |
| "项目的 token 预算超了" | 优化目标 2-3（项目模式） |

### 3 问验证框架

应用任何反馈前，每条都经过验证：

| 问题 | 目的 | 危险信号 |
|------|------|---------|
| **目标对齐：** 这是否服务于技能的核心目标？ | 防止范围漂移 | "这会把技能变成另一个东西" |
| **必要性：** 是否有实际缺陷，还是只是风格偏好？ | 防止不必要的变动 | "技能工作正常，我只是更喜欢另一种格式" |
| **副作用：** 是否可能引入复杂度或回归？ | 防止膨胀 | "为处理罕见边缘情况增加了 50 行" |

### 反馈流程

```
接收反馈
  → 识别目标技能
  → 如果是外部技能：以 forked- 前缀 fork
  → 读取技能，理解核心目标
  → 验证每条反馈（3 问框架）
  → 呈现改进方案 → 用户确认
  → 复制到工作版本
  → 对副本应用更改
  → A/B 评估（原始版 vs 优化版）
  → 用户决定：采用或丢弃
  → 优化（optimizing）调用 auditing 进行变更后验证
```

**规则：**
- 未经用户确认绝不应用反馈
- 对外部技能，必须先 fork（添加来源头）
- 变更后至多一次验证；若仍有问题，升级由用户决定 — 不要自动循环。按本技能协议时由**优化**触发重审；**审计**不会自动触发优化。

## 常见错误

| 错误 | 会出什么问题 | 如何避免 |
|------|------------|---------|
| 试图一次优化所有目标 | 变更分散，难以验证效果 | 选一个目标，度量、改进、验证 — 再处理下一个 |
| 把描述写成工作流摘要 | Agent 走描述捷径而不读完整 SKILL.md | 陈述触发*条件*（"Use when reviewing..."），而非*步骤*（"Scans structure, checks manifests..."） |
| 忽视引导技能的 token 预算 | 引导技能每次会话都加载，膨胀的代价波及所有地方 | 保持 `using-*` 在 200 行以内 — 这是 ROI 最高的 token 优化 |
| 不验证就应用用户反馈 | 风格偏好伪装成缺陷报告，导致不必要的变动 | 接受前让每条反馈过 3 问验证框架 |
| 基于反馈扩展技能范围 | 技能逐渐偏离原始职责 | 反馈应该改进技能*如何*工作，而非改变它*是什么* |
| 对单个技能运行全部 8 个目标 | 目标 5-8 需要项目上下文，在技能范围下无有用结果 | 让范围自动检测处理 — 单技能只适用目标 1-4 |
| 添加第三方技能不做安全审计 | 导入内容可能包含编码提示词、过量工具权限或网络调用 | 始终对导入的技能运行 `bundles-forge:auditing` — 参见 `references/third-party-integration.md` |
| 添加技能不更新 Integration 声明 | 工作流图不一致，触发 W10（不对称集成）发现 | 每个新技能连接需要对称的 `**Calls:**` 和 `**Called by:**` 声明 |
| 描述重写跳过 A/B 评估 | 改善一个触发器可能破坏另一个 | 修改现有触发短语时始终做 A/B 评估 — 纯添加性变更可以跳过 |

## FAQ

**Q：审计和优化的区别是什么？**

审计是纯诊断 — 检查、评分、出报告。它从不修改文件，也不调用优化。优化是改进驱动者 — 它读取审计报告（或你的目标），诊断需要修复什么，将内容变更委派给 authoring，并通过调用审计验证结果。简单来说：审计告诉你哪里有问题，优化来修复。

**Q：什么时候该用 optimizing，什么时候直接用 authoring？**

当你需要*诊断*时用 optimizing — 不确定具体该修什么，或者需要结构化的改进流程（含 A/B 评估和验证）。当你已经明确知道要写或改什么内容时直接用 authoring（如「把这个描述改成 X」）。

**Q：优化前必须先运行审计吗？**

不是必须的，但推荐。优化技能可以自行诊断，但提供审计报告让它有一个按优先级排列的发现列表可以处理。常见模式是：审计 → 审查报告 → 基于发现优化。

**Q：优化单个技能时哪些目标适用？**

目标 1-4（描述触发准确性、token 效率、渐进式加载、工作流链路完整性）加反馈迭代。目标 5-8 跳过，因为它们需要项目级上下文。在目标 4 中，单技能范围下仅 W9（占位章节）和 W10（不对称集成）适用。

**Q：优化后验证审计仍有问题怎么办？**

Agent 会将剩余问题呈现给你做手动决定 — 不会自动循环。你可以选择再跑一轮优化、直接调用 authoring 做特定修复、或接受当前状态。这防止了无限的优化-审计循环。

**Q：能直接优化 GitHub 上的项目而不用先克隆吗？**

可以。直接传 GitHub URL — 技能会自动执行浅克隆。这也适用于子目录 URL（如 `github.com/user/repo/tree/main/skills/my-skill`）和归档 URL（.zip/.tar.gz）。
