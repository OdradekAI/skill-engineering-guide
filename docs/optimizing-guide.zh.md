# 优化指南

[English](optimizing-guide.md)

面向用户的 bundle-plugin 及单个技能优化指南。涵盖范围检测、6 个优化目标、A/B 评估、反馈迭代和审计-优化循环。

## 概述

优化（optimizing）是迭代改进技能。与审计（评估）不同，优化采取行动 — 修复描述、减少 token、收紧工作流链路、处理用户反馈。

**核心原则：** 为 Agent 的体验而优化。每项改进都应该让技能更容易被发现、更快加载、更清晰可循。

> **权威来源：** 完整的执行协议（范围检测、目标详情、A/B 评估步骤、反馈验证）位于 `skills/optimizing/SKILL.md`。本指南帮助你理解*每个目标做什么*、*如何解读结果*、*何时使用哪种方法*。

---

## 范围检测

优化会自动检测你是在处理完整项目还是单个技能：

| 目标 | 检测方式 | 模式 | 适用目标 |
|------|---------|------|---------|
| 项目根目录 | 包含 `skills/` + `package.json` | 项目优化 | 全部 6 个目标 |
| 单个技能目录 | 包含 `SKILL.md`，无 `skills/` 子目录 | 技能优化 | 4 个目标 + 反馈 |

你不需要指定模式 — 技能会从你提供的路径自动检测。

### 输入来源

优化可以消费先前审计的报告。推荐工作流 — 先审计，再根据发现优化：

| 输入 | 来源 | 用途 |
|------|------|------|
| `audit-report` | `bundles-forge:auditing`（完整项目） | 所有 6 个目标的逐技能分解 |
| `skill-report` | `bundles-forge:auditing`（技能模式） | 技能优化的聚焦 4 类报告 |
| `workflow-report` | `bundles-forge:auditing`（工作流模式） | 目标 4 的 W1-W12 发现 |
| `user-feedback` | 直接来自用户 | 迭代过程的行为反馈 |

## 6 个优化目标

### 目标 1：技能描述触发准确性

**影响最大的优化。** 描述是技能发现的主要机制 — 当用户说话时，Agent 将意图与描述字段匹配。

**关键规则：** 描述必须陈述*触发条件*，而非*工作流摘要*。

| | 差（工作流摘要） | 好（触发条件） |
|---|---|---|
| **模式** | "Use for auditing — scans structure, checks manifests, scores categories, generates report" | "Use when reviewing a bundle-plugin for structural issues, version drift, or before release" |
| **问题** | Agent 走描述捷径而不读完整 SKILL.md | Agent 读取完整技能获取执行细节 |
| **结果** | 跳过步骤、不完整执行 | 按设计完整执行 |

**其他规则：**
- 必须以 "Use when..." 开头
- 保持在 250 字符以内（超过会在技能列表中截断）
- 包含具体的症状、场景和上下文
- 不要提及技能的内部步骤

#### 如何验证描述质量

运行 linter 捕获机械性问题：

```bash
python scripts/lint_skills.py <project-root>
```

检查项 Q1-Q12 覆盖描述格式、长度和反模式。对于*行为*质量（正确的提示是否触发正确的技能），使用 A/B 评估。

### 目标 2：Token 效率

频繁加载的技能中每个 token 都会占用每次会话的上下文预算。这对引导技能（每次会话加载）和常触发的技能影响最大。

**目标：**
- SKILL.md 正文 < 500 行
- 引导技能（`using-*`）< 200 行
- 将重量级参考资料移至 `references/`

**技巧：**

| 技巧 | 示例 |
|------|------|
| 交叉引用代替重复 | `See bundles-forge:authoring` 代替复制规则 |
| 一个优秀示例胜过三个平庸示例 | 删除教授相同概念的冗余示例 |
| 将标志文档指向 --help | 引用 `python scripts/lint_skills.py --help` 代替列出所有标志 |
| 消除项目内冗余 | 不要重复其他技能 `references/` 中已有的内容 |

### 目标 3：渐进式加载

三级加载系统确保最小上下文占用：

| 级别 | 何时加载 | 预算 |
|------|---------|------|
| 元数据（name + description） | 始终在上下文中 | ~100 词 |
| SKILL.md 正文 | 技能触发时 | < 500 行 |
| 参考文件（`references/`） | 按需 | 无限制 |

**何时提取到 references/：**
- SKILL.md 接近 500 行
- 仅在执行时需要的表格或检查清单（不是理解技能用途所需的）
- Agent 逐字复制的模板内容

### 目标 4：工作流链路完整性

消费工作流审计的 W1-W12 发现。如果没有工作流报告：

```bash
python scripts/audit_workflow.py <project-root>
python scripts/audit_workflow.py --focus-skills skill-a,skill-b <root>
```

**修复优先级指南：**

| 发现 | 含义 | 修复方式 |
|------|------|---------|
| W1（未声明循环） | 两个技能互相调用但循环未声明 | 如果有意则在 `## Integration` 中添加 `<!-- cycle:a,b -->`，否则重构 |
| W2（不可达技能） | 技能存在但没有链路指向它 | 添加到引导路由，或声明 `Called by: user directly` |
| W3/W4（缺失 I/O） | 终端技能没有 `## Outputs`，或被引用的技能没有 `## Inputs` | 添加带制品 ID 的章节 |
| W5（制品 ID 不匹配） | 上游 `## Outputs` 和下游 `## Inputs` 使用不同名称 | 对齐反引号包裹的制品 ID |
| W9（占位章节） | Inputs/Outputs 存在但为空或通用 | 编写有意义的语义描述 |
| W10（不对称集成） | 技能 A 说它调用 B，但 B 没说被 A 调用 | 添加缺失的 `**Called by:**` 声明 |

### 目标 5：平台覆盖（仅项目）

识别项目尚未支持的平台。添加新平台请调用 `bundles-forge:porting` — 优化技能本身不生成平台适配器。

### 目标 6：安全修复（仅项目）

修复 `bundles-forge:auditing` 类别 10 的安全发现。常见修复：

| 发现 | 修复 |
|------|------|
| Hook 脚本发起网络调用 | 移除或添加注释说明理由 |
| OpenCode 插件能力过大 | 限定到声明的需求范围 |
| Agent 提示词缺少范围约束 | 添加显式边界 |
| SKILL.md 包含编码/混淆内容 | 剥离或替换为明文 |

## A/B 评估

A/B 评估是描述变更和反馈驱动改进的核心质量保障机制。它并排比较原始版与优化版。

### 工作原理

```
1. 将技能复制到工作版本（<skill-name>-optimized/）
2. 仅对副本应用更改（不要先覆盖原始版）
3. 创建 5+ 个应该触发此技能的真实测试提示词
4. 并行派遣两个 evaluator agent：
   - Evaluator A："original" 标签 → 用原始技能测试
   - Evaluator B："optimized" 标签 → 用优化技能测试
5. 比较结果 → 呈现给用户
6. 用户决定：采用优化版或丢弃
```

### 比较指标

| 指标 | 含义 |
|------|------|
| **触发率** | 多少提示词正确激活了技能？ |
| **假阴性** | 优化版是否遗漏了原始版捕获的场景？ |
| **假阳性** | 任一版本是否在不相关的提示词上误触发？ |
| **步骤准确性** | Agent 是否遵循了所有步骤，还是走了捷径？ |

### 何时跳过 A/B

| 场景 | 跳过？ | 理由 |
|------|:---:|------|
| 纯添加性变更（新触发短语，无修改） | 是 | 简单验证即可 |
| 结构修复（缺失章节、断裂引用） | 是 | 不是行为变更 |
| 修改现有触发器的描述重写 | **否** | 必须验证无回归 |
| 反馈驱动的行为变更 | **否** | 必须比较新旧行为 |

### 链式 A/B 评估

用于工作流过渡（不是单个描述），使用链式评估：

1. 定义一个真实的端到端场景
2. 以 "chain" 标签和有序技能列表派遣 evaluator
3. 审查每个交接点的过渡质量评级
4. 关注 "broken" 交接 — 表示缺失制品或不清晰的指令

**何时使用链式评估：** 修改 Inputs/Outputs 后、向链路添加技能后、或 W1-W12 发现指示工作流问题时。

### 子代理回退

当子代理派遣不可用时，两个选项：

| 回退方式 | 方法 | 权衡 |
|---------|------|------|
| 顺序内联 | 按 `agents/evaluator.md` 协议内联执行，随机化顺序 | 较慢，可能有顺序偏差 |
| 跳过 A/B | 直接应用变更并简单验证 | 较快，无对比数据 |

由用户选择使用哪种回退。

## 反馈迭代

当用户报告技能触发了但产出错误结果时，反馈流程提供结构化的迭代。

### 反馈分类

| 用户说 | 操作 |
|--------|------|
| "这个技能触发了但结果不对" | 反馈迭代 |
| "步骤顺序不对" | 反馈迭代 |
| "描述格式不符合规范" | 优化目标 1-3 |
| "项目的 token 预算超了" | 优化目标 2-3（项目模式） |

### 3 问验证框架

应用任何反馈前，每条都经过验证：

| 问题 | 目的 | 危险信号 |
|------|------|---------|
| **目标对齐：** 这是否服务于技能的核心目标？ | 防止范围漂移 | "这会把技能变成另一个东西" |
| **必要性：** 是否有实际缺陷，还是只是风格偏好？ | 防止不必要的变动 | "技能工作正常，我只是更喜欢另一种格式" |
| **副作用：** 是否可能引入复杂度或回归？ | 防止膨胀 | "为处理罕见边缘情况增加了 50 行" |

### 反馈流程

```
接收反馈
  → 识别目标技能
  → 如果是外部技能：以 forked- 前缀 fork
  → 读取技能，理解核心目标
  → 验证每条反馈（3 问框架）
  → 呈现改进方案 → 用户确认
  → 复制到工作版本
  → 对副本应用更改
  → A/B 评估（原始版 vs 优化版）
  → 用户决定：采用或丢弃
  → 运行 auditing 进行变更后验证
```

**规则：**
- 未经用户确认绝不应用反馈
- 对外部技能，必须先 fork（添加来源头）
- 变更后只运行一个审计周期（不循环）

## 审计-优化循环

优化和审计构成一个有意设计的循环：

```
auditing 发现问题 → optimizing 修复 → auditing 验证修复
                                        ↑ 在此停止（一个循环）
```

**重要：** 循环只运行一次。如果重审仍有问题，交给用户手动决定 — 不要无限循环。

### 推荐工作流

1. 运行 `bundles-forge:auditing`（完整、技能或工作流模式）
2. 审查报告 — 优先处理严重发现
3. 以审计报告为输入运行 `bundles-forge:optimizing`
4. 修复后，auditing 自动运行验证
5. 审查验证报告 — 如仍有问题，手动决定
