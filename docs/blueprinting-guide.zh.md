# 蓝图规划指南

[English](blueprinting-guide.md)

面向用户的 bundle-plugin 规划指南。涵盖场景选择、设计访谈、自动化创建流水线和常见陷阱。

## 概述

蓝图规划（blueprinting）是**新项目的编排器**：它负责结构化设计访谈，并在你批准设计文档之后协调完整的创建流水线 — 脚手架、内容撰写、工作流串联和初始质量检查。它在生成任何代码或项目结构之前，将模糊的想法转化为具体的设计文档。

**为什么重要：** 5 分钟的蓝图规划可以节省数小时的返工。跳过此步骤是导致 bundle-plugin 结构混乱、需要完全重构的首要原因。

> **权威来源：** 完整的执行协议（访谈问题、场景分析步骤、设计文档模板）位于 `skills/blueprinting/SKILL.md`。本指南帮助你决定*走哪条路*、*预期什么结果* — 技能本身负责执行。

## 快速开始

创建新 bundle-plugin 的最快方式：

1. **调用** `/bundles-blueprint`（或请求 agent 规划一个新的 bundle-plugin）
2. **回答**结构化访谈 — 每次一个问题
3. **审查并批准** agent 生成的设计文档
4. **agent 接管** — 自动运行 scaffolding → authoring → 工作流设计 → auditing

两个需要提前做出的关键决策：

- **快速模式 vs 自适应模式？** 快速模式用于快速打包几个独立技能（仅 Claude Code，无引导技能）。自适应模式用于有工作流链路的编排式多技能项目。agent 会在第一个问题中询问你。
- **已有项目？** 蓝图规划只用于创建*新*项目。要添加技能、重构工作流或改进现有项目，请使用 `/bundles-optimize` — 参见[优化指南](optimizing-guide.zh.md)中的目标 7。

详细说明请继续阅读。

## 选择场景

蓝图规划有三个入口。选择与你情况匹配的：

| 场景 | 起点 | 适用场景 | 输出 |
|------|------|---------|------|
| **A：新项目** | 没有现有代码 | 从零开始，有一个想法 | 完整设计文档 |
| **B：拆分** | 一个大型技能 | SKILL.md 已经过于复杂 | 拆分方案 → 设计文档 |
| **C：组合** | 多个现有技能 | 想将独立技能组合成工作流 | 组合方案 → 设计文档 |

> **想向现有项目添加技能？** 那是优化而非蓝图规划。使用 `/bundles-optimize` — 参见[优化指南](optimizing-guide.zh.md)中的目标 7（技能与工作流重构）。

### 决策流程图

```
你有现成的 bundle-plugin 项目吗？
  ├─ 是 → 你可能需要 optimizing（添加技能、重构）或 auditing，不是 blueprinting
  └─ 否 → 你有现成的技能吗？
           ├─ 是 → 是一个需要拆分的大型技能吗？
           │        ├─ 是 → 场景 B（拆分）
           │        └─ 否 → 多个技能要组合？
           │                  ├─ 是 → 场景 C（组合）
           │                  └─ 否 → 场景 A（新项目）
           └─ 否 → 场景 A（新项目）
```

## 设计访谈

**最常见的路径（场景 A）。** 你有一个 bundle-plugin 的想法，想要正确地规划它。

### 预期流程

技能会运行结构化访谈，每次问一个问题：

| # | 问题 | 为什么重要 |
|---|------|-----------|
| 1 | 项目复杂度 | 决定访谈深度（快速模式 vs 自适应模式） |
| 2 | 项目名称 | 成为所有地方的目录名、包名和插件 ID |
| 3 | 目标平台 | 决定生成哪些清单文件 |
| 4 | 技能清单 | 定义插件有哪些能力 |
| 4a | 技能可见性 | 入口技能 vs 内部技能 — 影响命令和描述 |
| 5 | 工作流链路 | 决定技能如何连接以及引导内容 |
| 6 | 引导策略 | 是否在会话启动时自动注入技能感知 |
| 7 | 高级组件 | MCP 服务器、LSP 服务器、bin/ 可执行文件、输出风格 |

问题 4a、5、6、7 仅在**自适应模式**（编排式多技能项目）中提问。快速模式（快速打包）会跳过。

### 快速模式 vs 自适应模式

| | 快速模式 | 自适应模式 |
|---|---|---|
| **适用场景** | 将独立技能打包分发 | 构建编排式工作流 |
| **访谈深度** | 4 个问题（名称、平台、技能、完成） | 7+ 个问题并有条件追问 |
| **默认平台** | 仅 Claude Code | 用户自选 |
| **引导技能** | 跳过（不需要） | 3+ 技能时推荐 |
| **命令** | 不生成 | 入口技能获得对应命令 |
| **时长** | 2-3 分钟 | 5-10 分钟 |

**经验法则：** 如果技能形成链路（A 的输出供 B 使用），用自适应模式。如果是独立工具集，快速模式就够了。

### 回答建议

- **项目名称：** 使用 kebab-case（`my-dev-tools`，不是 `myDevTools`）。这个名称会出现在所有地方。
- **平台：** 从你实际使用的平台开始，后续可以用 `bundles-forge:scaffolding` 添加其他平台。
- **技能清单：** 粗略列表就行，后续可以添加。
- **工作流链路：** 画出来 — "skill-a → skill-b → skill-c"。如果技能独立，直接说明。

## 高级场景

大多数用户走标准访谈（场景 A）。以下两个场景处理特殊起点 — 两者都会汇入同一访谈来完成项目细节。

### 场景 B：拆分（分解大型技能）

**你有一个过于复杂的 SKILL.md** — 职责太多、分支逻辑复杂、或接近 500 行限制。

**需要拆分的信号：**

| 信号 | 含义 |
|------|------|
| SKILL.md > 300 行且有多个"如果 X 就做这个；如果 Y 就做那个"的分支 | 每个主要分支都是独立技能的候选 |
| 有独立输入/输出格式的章节 | 这些是不同的职责 |
| 某些步骤在部分情况下可以完全跳过 | 可选步骤应该是链路中的独立技能 |
| 特定子任务的大量参考资料 | 子任务应该是有自己 `references/` 的独立技能 |

**预期流程：**

1. 技能读取你现有的 SKILL.md，映射其职责
2. 根据上述信号识别自然分割点
3. 提出拆分方案：哪些部分成为技能，哪些成为共享资源
4. 你批准后，转入标准访谈完成剩余项目细节

**常见模式：** 分支拆分（if/else 分支 → 独立技能）、流水线提取（顺序阶段 → 链式技能）、参考资料提取（内联表格 → `references/` 文件）。并非每个大型技能都需要拆分 — 如果技能虽然很长但有单一明确的职责，提取到 `references/` 即可。

### 场景 C：组合（合并多个技能）

**你有多个独立技能，想让它们作为协调的 bundle-plugin 协作。**

**预期流程：**

1. 技能清点所有候选技能（来源、结构、质量、许可证）
2. 检查兼容性（命名冲突、职责重叠、风格一致性）
3. 对第三方技能，询问集成意图：
   - **原样打包** — 不修改直接捆绑（添加来源标注）
   - **集成到工作流** — 适配描述、交叉引用和交接
4. 设计编排（工作流链路、粘合技能、共享资源、引导）
5. 你批准后，转入标准访谈完成剩余细节

**第三方技能处理：**

| 步骤 | 必需？ | 目的 |
|------|:---:|------|
| 许可证检查 | 是 | 确保许可证兼容性 |
| 安全审计 | 是 | 对导入内容运行 `bundles-forge:auditing` |
| 来源标注 | 是 | 在复制的 SKILL.md 中添加来源头 |
| 描述重写 | 仅工作流集成 | 将触发条件适配到新上下文 |
| 交叉引用重写 | 仅工作流集成 | 将 `old-project:skill` 改为 `new-project:skill` |

**绝不自动安装存在未解决严重安全发现的第三方技能。**

## 设计文档

所有场景都产出如下结构的设计文档：

```markdown
## Bundle-Plugin Design: <project-name>

**Mode:** quick / adaptive
**Platforms:** <list>
**Bootstrap:** yes/no

### Skills
| Name | Purpose | Type | Visibility | Dependencies |
|------|---------|------|------------|--------------|

### Workflow Chain
<description or "independent">

### Advanced Components
<list or "none">

### Third-Party Sources (if applicable)
| Skill | Source | License | Integration Intent |
|-------|--------|---------|-------------------|

### Notes
<special requirements or constraints>
```

**批准前检查清单：**
- [ ] 所有技能名使用 kebab-case
- [ ] 工作流依赖已映射（或标记为独立）
- [ ] 平台选择与实际使用一致
- [ ] 第三方技能已注明许可证和安全审计情况

## 流水线：批准后的自动流程

设计文档获批后，**蓝图规划会编排**四阶段流水线。这不是仅向脚手架做一次自动交接；编排器按顺序推进各阶段：

| 阶段 | 执行内容 | 目的 |
|------|----------|------|
| **阶段 1** | `bundles-forge:scaffolding` | 生成项目结构与各平台清单 |
| **阶段 2** | `bundles-forge:authoring` | 撰写 **SKILL.md** 与 **agents/*.md** — 由**蓝图规划**在脚手架之后调用（**不由** scaffolding 调用） |
| **阶段 3** | 工作流设计 | 按设计将技能串联（链路、引导、命令等） |
| **阶段 4** | `bundles-forge:auditing` | 在进一步迭代前做质量与安全校验 |

若后续需要增加平台，可再次使用 `bundles-forge:scaffolding`。

Agent 会携带你已批准的设计文档贯穿上述阶段。**authoring** 作为该流水线的一环由**蓝图规划**调用；脚手架不会调用 authoring。

## 常见问题

**Q：Agent 跳过访谈直接进入了脚手架，怎么回事？**

技能未加载，或者你说了类似"直接生成"的话绕过了访谈。显式调用 `/bundles-blueprint` 以确保完整访谈流程运行。

**Q：访谈问的问题太多，我的项目很简单。**

你处于自适应模式。在问题 1 回答"快速打包"（或"快速模式"）即可使用仅 4 个问题的简短访谈流程。

**Q：访谈太短了，跳过了重要问题。**

你处于快速模式。重新开始，在问题 1 回答"编排式多技能"以获得完整的自适应模式访谈和条件追问。

**Q：设计文档列出了错误的平台。**

在批准前编辑设计文档 — agent 会展示给你审查。你可以在批准前修改平台、技能名或任何其他字段。

**Q：第三方技能安全审计失败了。**

修复导入技能中的安全问题，或将其从组合中排除。绝不批准包含存在未解决严重安全发现的第三方技能的设计。

**Q：不确定该选哪个场景。**

参照"选择场景"章节的决策流程图。如果不确定，从场景 A（新项目）开始 — 如果实际需要拆分或组合，agent 可以适配调整。

**Q：我想向现有项目添加技能，而不是创建新项目。**

使用 `/bundles-optimize` — 目标 7（技能与工作流重构）处理在现有项目中添加、替换和重组技能。蓝图规划仅用于创建新项目。

**Q：蓝图规划和脚手架有什么区别？**

蓝图规划是*规划*阶段 — 它访谈你并产出设计文档。脚手架是*执行*阶段 — 它读取设计并生成实际的项目文件。对于在现有项目上的平台适配，你可以跳过蓝图规划，直接调用脚手架。

**Q：我可以跳过蓝图规划直接 scaffold 吗？**

可以，但仅限于平台适配（在现有项目上添加/移除平台支持）。对于新项目，跳过访谈意味着 agent 需要猜测你的需求 — 这会导致返工。5 分钟的蓝图规划几乎总是值得的。
