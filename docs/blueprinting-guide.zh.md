# 蓝图规划指南

[English](blueprinting-guide.md)

面向用户的 bundle-plugin 规划指南。涵盖全部四种入口场景、决策指引、真实用例和常见陷阱。

## 概述

蓝图规划（blueprinting）是 bundle-plugin 生命周期的第一个技能。它通过结构化访谈将模糊的想法转化为具体的设计文档 — 在生成任何代码或项目结构之前。

**为什么重要：** 5 分钟的蓝图规划可以节省数小时的返工。跳过此步骤是导致 bundle-plugin 结构混乱、需要完全重构的首要原因。

> **权威来源：** 完整的执行协议（访谈问题、场景分析步骤、设计文档模板）位于 `skills/blueprinting/SKILL.md`。本指南帮助你决定*走哪条路*、*预期什么结果* — 技能本身负责执行。

---

## 选择场景

蓝图规划有四个入口。选择与你情况匹配的：

| 场景 | 起点 | 适用场景 | 输出 |
|------|------|---------|------|
| **A：新项目** | 没有现有代码 | 从零开始，有一个想法 | 完整设计文档 |
| **B：拆分** | 一个大型技能 | SKILL.md 已经过于复杂 | 拆分方案 → 设计文档 |
| **C：组合** | 多个现有技能 | 想将独立技能组合成工作流 | 组合方案 → 设计文档 |
| **D：集成** | 现有 bundle-plugin 项目 | 想向已有项目添加技能 | 轻量级集成方案 |

### 决策流程图

```
你有现成的 bundle-plugin 项目吗？
  ├─ 是 → 你要向它添加新技能吗？
  │        ├─ 是 → 场景 D（集成）
  │        └─ 否 → 你可能需要 auditing 或 optimizing，不是 blueprinting
  └─ 否 → 你有现成的技能吗？
           ├─ 是 → 是一个需要拆分的大型技能吗？
           │        ├─ 是 → 场景 B（拆分）
           │        └─ 否 → 多个技能要组合？
           │                  ├─ 是 → 场景 C（组合）
           │                  └─ 否 → 场景 A（新项目）
           └─ 否 → 场景 A（新项目）
```

## 场景 A：从零新建

**最常见的路径。** 你有一个 bundle-plugin 的想法，想要正确地规划它。

### 预期流程

技能会运行结构化访谈，每次问一个问题：

| # | 问题 | 为什么重要 |
|---|------|-----------|
| 0 | 项目复杂度 | 决定访谈深度（精简模式 vs 智能模式） |
| 1 | 项目名称 | 成为所有地方的目录名、包名和插件 ID |
| 2 | 目标平台 | 决定生成哪些清单文件 |
| 3 | 技能清单 | 定义插件有哪些能力 |
| 3b | 技能可见性 | 入口技能 vs 内部技能 — 影响命令和描述 |
| 4 | 工作流链路 | 决定技能如何连接以及引导内容 |
| 5 | 引导策略 | 是否在会话启动时自动注入技能感知 |
| 6 | 高级组件 | MCP 服务器、LSP 服务器、bin/ 可执行文件、输出风格 |

问题 3b、4、5、6 仅在**智能模式**（编排式多技能项目）中提问。精简模式（快速打包）会跳过。

### 精简模式 vs 智能模式

| | 精简模式 | 智能模式 |
|---|---|---|
| **适用场景** | 将独立技能打包分发 | 构建编排式工作流 |
| **访谈深度** | 4 个问题（名称、平台、技能、完成） | 7+ 个问题并有条件追问 |
| **引导技能** | 跳过（不需要） | 3+ 技能时推荐 |
| **命令** | 不生成 | 入口技能获得对应命令 |
| **时长** | 2-3 分钟 | 5-10 分钟 |

**经验法则：** 如果技能形成链路（A 的输出供 B 使用），用智能模式。如果是独立工具集，精简模式就够了。

### 回答建议

- **项目名称：** 使用 kebab-case（`my-dev-tools`，不是 `myDevTools`）。这个名称会出现在所有地方。
- **平台：** 从你实际使用的平台开始，后续可以用 `bundles-forge:porting` 添加其他平台。
- **技能清单：** 粗略列表就行，后续可以添加。
- **工作流链路：** 画出来 — "skill-a → skill-b → skill-c"。如果技能独立，直接说明。

## 场景 B：拆分（分解大型技能）

**你有一个过于复杂的 SKILL.md** — 职责太多、分支逻辑复杂、或接近 500 行限制。

### 需要拆分的信号

| 信号 | 含义 |
|------|------|
| SKILL.md > 300 行且有多个"如果 X 就做这个；如果 Y 就做那个"的分支 | 每个主要分支都是独立技能的候选 |
| 有独立输入/输出格式的章节 | 这些是不同的职责 |
| 某些步骤在部分情况下可以完全跳过 | 可选步骤应该是链路中的独立技能 |
| 特定子任务的大量参考资料 | 子任务应该是有自己 `references/` 的独立技能 |

### 预期流程

1. 技能读取你现有的 SKILL.md，映射其职责
2. 根据上述信号识别自然分割点
3. 提出拆分方案：哪些部分成为技能，哪些成为共享资源
4. 你批准后，转入场景 A 的访谈完成剩余项目细节

### 常见拆分模式

| 模式 | 拆分前 | 拆分后 |
|------|--------|--------|
| **分支拆分** | 一个技能有 3 个主要 if/else 分支 | 3 个技能各处理一个分支，由父技能编排 |
| **流水线提取** | 一个技能有顺序阶段 | 阶段技能通过工作流链接 |
| **参考资料提取** | 一个技能有内联参考表 | 核心技能 + `references/` 中的参考文件（不是独立技能） |

**重要：** 并非每个大型技能都需要拆分。如果技能虽然很长但有单一明确的职责，提取到 `references/` 即可，不必拆分成多个技能。

## 场景 C：组合（合并多个技能）

**你有多个独立技能，想让它们作为协调的 bundle-plugin 协作。**

### 预期流程

1. 技能清点所有候选技能（来源、结构、质量、许可证）
2. 检查兼容性（命名冲突、职责重叠、风格一致性）
3. 对第三方技能，询问集成意图：
   - **原样打包** — 不修改直接捆绑（添加来源标注）
   - **集成到工作流** — 适配描述、交叉引用和交接
4. 设计编排（工作流链路、粘合技能、共享资源、引导）
5. 你批准后，转入场景 A 的访谈完成剩余细节

### 第三方技能处理

| 步骤 | 必需？ | 目的 |
|------|:---:|------|
| 许可证检查 | 是 | 确保许可证兼容性 |
| 安全审计 | 是 | 对导入内容运行 `bundles-forge:auditing` |
| 来源标注 | 是 | 在复制的 SKILL.md 中添加来源头 |
| 描述重写 | 仅意图 B | 将触发条件适配到新上下文 |
| 交叉引用重写 | 仅意图 B | 将 `old-project:skill` 改为 `new-project:skill` |

**绝不自动安装存在未解决严重安全发现的第三方技能。**

## 场景 D：集成（向现有项目添加技能）

**你的 bundle-plugin 已经在工作，你想向其中添加新技能。** 这比场景 A-C 更轻量 — 产出集成方案而非完整设计文档。

### 预期流程

1. 技能读取现有项目（技能、工作流图、引导路由）
2. 清点要添加的新技能（与场景 C 相同的分析）
3. 对照*现有项目*（而非空白）检查兼容性
4. 设计插入点：新技能在哪里连接到工作流？
5. 你批准后，应用变更并建议运行聚焦式工作流审计

### 集成方案 vs 设计文档

| | 设计文档（A/B/C） | 集成方案（D） |
|---|---|---|
| **范围** | 整个项目 | 仅变更部分 |
| **包含** | 名称、平台、全部技能、完整工作流 | 新技能、插入点、更新的边 |
| **后续** | `scaffolding` → `authoring` | 直接修改 + 工作流审计 |
| **耗时** | 完整生命周期 | 快速迭代 |

### 集成后验证

集成后务必运行聚焦式工作流审计：

```bash
python scripts/audit_workflow.py --focus-skills new-skill-a,new-skill-b .
```

这会验证新技能正确集成且不破坏现有工作流链路。

## 设计文档

除场景 D 外，所有场景都产出如下结构的设计文档：

```markdown
## Bundle-Plugin Design: <project-name>

**Mode:** minimal / intelligent
**Platforms:** <list>
**Bootstrap:** yes/no

### Skills
| Name | Purpose | Type | Visibility | Dependencies |
|------|---------|------|------------|--------------|

### Workflow Chain
<description or "independent">

### Advanced Components
<list or "none">

### Third-Party Sources (if applicable)
| Skill | Source | License | Integration Intent |
|-------|--------|---------|-------------------|

### Notes
<special requirements or constraints>
```

**批准前检查清单：**
- [ ] 所有技能名使用 kebab-case
- [ ] 工作流依赖已映射（或标记为独立）
- [ ] 平台选择与实际使用一致
- [ ] 第三方技能已注明许可证和安全审计情况

## 蓝图规划之后

| 后续步骤 | 触发方式 |
|---------|---------|
| `bundles-forge:scaffolding` | 设计文档批准后（自动链式调用） |
| `bundles-forge:authoring` | 脚手架生成项目结构后 |
| `bundles-forge:porting` | 如果选择了多平台目标 |

到 scaffolding 的过渡是自动的 — Agent 会携带你批准的设计文档调用它。

## 故障排除

| 问题 | 原因 | 修复 |
|------|------|------|
| Agent 跳过访谈直接进入脚手架 | 技能未加载或用户说了"直接生成" | 显式调用 `/bundles-blueprint` |
| 访谈问太多问题 | 对简单项目使用了智能模式 | 在问题 0 回答"快速打包" |
| 访谈问太少问题 | 对复杂项目使用了精简模式 | 重新开始并在问题 0 回答"编排式多技能" |
| 设计文档平台选错了 | 平台问题回答太快 | 在批准前编辑设计文档 |
| 第三方技能安全审计失败 | 导入的技能有风险模式 | 修复问题或排除该技能 |
| 不确定选哪个场景 | 多个场景似乎都适用 | 参照上面的决策流程图 |
