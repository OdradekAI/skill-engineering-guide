# 蓝图规划指南

[English](blueprinting-guide.md)

面向用户的 bundle-plugin 规划指南。涵盖三阶段访谈（需求探索、架构设计、设计审核）、自动化创建流水线和常见陷阱。

## 概述

蓝图规划（blueprinting）是**新项目的编排器**：它运行三阶段结构化访谈 — 首先理解你要构建*什么*以及*为什么*，然后设计技术架构，最后生成设计文档供你审批。审批通过后，它协调完整的创建流水线：脚手架生成、内容撰写、工作流串联和初始质量检查。

**为什么重要：** 5 分钟的需求探索可以节省数小时的返工。跳过此步骤是导致 bundle-plugin 结构混乱、需要完全重构的首要原因。

> **权威来源：** 完整的执行协议（访谈阶段、对话策略、设计文档模板）位于 `skills/blueprinting/SKILL.md`。本指南帮助你决定*走哪条路*、*预期什么结果* — 技能本身负责执行。

## 快速开始

创建新 bundle-plugin 的最快方式：

1. **调用** `/bundles-blueprint`（或请求 agent 规划一个新的 bundle-plugin）
2. **回答**三阶段访谈 — 先谈需求，再谈架构，每次一个问题
3. **审查并批准** agent 生成的设计文档（含自审检查）
4. **agent 接管** — 自动运行 scaffolding → authoring → 工作流设计 → auditing

关键须知：

- **需求优先。** agent 会先问你要解决什么问题、给谁用，*然后*才问技术问题。即使是简单项目，也只需 2 个问题。
- **agent 推荐，你决定。** 在关键决策点（快速/自适应模式、技能拆分、平台选择），agent 会提供 2-3 个方案及优劣分析和推荐。
- **agent 会在必要时推回。** 如果你的回答互相矛盾，或请求的范围与问题不成比例，agent 会指出问题并提出更简单的替代方案。这是设计行为。
- **已有项目？** 蓝图规划只用于创建*新*项目。要添加技能、重构工作流或改进现有项目，请使用 `/bundles-optimize` — 参见[优化指南](optimizing-guide.zh.md)中的目标 7。

详细说明请继续阅读。

## 选择场景

蓝图规划有三个入口。选择与你情况匹配的：

| 场景 | 起点 | 适用场景 | 输出 |
|------|------|---------|------|
| **A：新项目** | 没有现有代码 | 从零开始，有一个想法 | 完整设计文档 |
| **B：拆分** | 一个大型技能 | SKILL.md 已经过于复杂 | 拆分方案 → 设计文档 |
| **C：组合** | 多个现有技能 | 想将独立技能组合成工作流 | 组合方案 → 设计文档 |

> **想向现有项目添加技能？** 那是优化而非蓝图规划。使用 `/bundles-optimize` — 参见[优化指南](optimizing-guide.zh.md)中的目标 7（技能与工作流重构）。

### 决策流程图

```
你有现成的 bundle-plugin 项目吗？
  ├─ 是 → 你可能需要 optimizing（添加技能、重构）或 auditing，不是 blueprinting
  └─ 否 → 你有现成的技能吗？
           ├─ 是 → 是一个需要拆分的大型技能吗？
           │        ├─ 是 → 场景 B（拆分）
           │        └─ 否 → 多个技能要组合？
           │                  ├─ 是 → 场景 C（组合）
           │                  └─ 否 → 场景 A（新项目）
           └─ 否 → 场景 A（新项目）
```

## 三阶段访谈

### 阶段 1：需求探索

**agent 先理解你的问题，而不是你的技术偏好。**

| # | 问题 | 为什么重要 |
|---|------|-----------|
| 1 | 问题场景 | 当前存在什么差距，人们怎么解决的 |
| 2 | 目标用户 | 影响平台选择、复杂度和文档风格 |
| 3 | 核心能力 | 识别不可或缺的技能 vs 锦上添花 |
| 4 | 使用流程 | 揭示工作流依赖和入口点 |
| 5 | 现有替代方案 | 明确差异化（仅在相关时提问） |

对于**快速模式**项目（简单技能打包），agent 只问问题 1 和 2 就进入下一阶段。

收集完答案后，agent 会复述理解并请你确认，确认后才进入架构设计。

### 阶段 2：架构设计

**agent 基于你的需求给出技术推荐。**

| # | 问题 | 运作方式 |
|---|------|---------|
| — | 假设门控 | agent 列出从阶段 1 推导的关键假设 — 你在架构开始前确认或纠正 |
| 1 | 项目复杂度 | agent 基于阶段 1 *推荐*快速/自适应模式 — 你确认 |
| 2 | 项目名称 | 到处使用的 kebab-case 标识符 |
| 3 | 目标平台 | agent 基于目标用户推荐 — 你确认或调整 |
| 4 | 技能清单 | agent *提议*拆分方案并附优劣分析 — 你确认或调整 |
| 4a | 技能可见性 | 入口技能 vs 内部技能 — 影响命令和描述（仅自适应模式） |
| 5 | 工作流链路 | 技能如何连接（仅自适应模式） |
| 6 | 引导策略 | 是否自动注入技能感知（仅自适应模式） |
| 7 | 高级组件 | MCP 服务器、LSP 服务器、bin/ 可执行文件、输出风格（仅自适应模式，仅在你主动提及相关需求时） |

问题 4a、5、6、7 仅在**自适应模式**中提问。快速模式跳过。

### 阶段 3：设计文档与审核

1. agent 将访谈内容整理为设计文档
2. 执行自审（占位符扫描、一致性检查、范围检查、模糊度检查）
3. 展示文档供你审批
4. 你可以要求修改 — agent 更新并重新自审
5. 只有你明确批准后，创建流水线才会启动

### 快速模式 vs 自适应模式

| | 快速模式 | 自适应模式 |
|---|---|---|
| **适用场景** | 将独立技能打包分发 | 构建编排式工作流 |
| **需求探索** | 2 个问题（问题 + 用户） | 5 个问题（完整需求探索） |
| **架构深度** | 5 个步骤（假设声明、模式、名称、平台、技能） | 8+ 个问题并有条件追问 |
| **默认平台** | 仅 Claude Code | 用户自选 |
| **引导技能** | 跳过（不需要） | 3+ 技能时推荐 |
| **命令** | 不生成 | 入口技能获得对应命令 |
| **时长** | 3-5 分钟 | 8-15 分钟 |

**经验法则：** 如果技能形成链路（A 的输出供 B 使用），用自适应模式。如果是独立工具集，快速模式就够了。agent 会推荐模式 — 你不需要提前选择。

### 回答建议

- **问题场景：** 具体描述痛点。"手动做 X 很烦"比"我想做个 X 工具"更有用。
- **目标用户：** 提及技能水平和工作流上下文。"使用 Claude Code 写 Python 的后端开发者"比"开发者"更有用。
- **项目名称：** 使用 kebab-case（`my-dev-tools`，不是 `myDevTools`）。这个名称会出现在所有地方。
- **平台：** 从你实际使用的平台开始，后续可以用 `bundles-forge:scaffolding` 添加其他平台。
- **当 agent 提出多个方案时：** 仔细看优劣分析。如果都不合适，直接说明 — agent 会调整。

## 高级场景

大多数用户走标准访谈（场景 A）。以下两个场景处理特殊起点 — 两者都会汇入同一三阶段访谈来完成项目细节。

### 场景 B：拆分（分解大型技能）

**你有一个过于复杂的 SKILL.md** — 职责太多、分支逻辑复杂、或接近 500 行限制。拆分会产出一个**新**项目。要在现有项目内重构技能，请使用 `/bundles-optimize`（目标 7）。

**需要拆分的信号：**

| 信号 | 含义 |
|------|------|
| SKILL.md > 300 行且有多个"如果 X 就做这个；如果 Y 就做那个"的分支 | 每个主要分支都是独立技能的候选 |
| 有独立输入/输出格式的章节 | 这些是不同的职责 |
| 某些步骤在部分情况下可以完全跳过 | 可选步骤应该是链路中的独立技能 |
| 特定子任务的大量参考资料 | 子任务应该是有自己 `references/` 的独立技能 |

**预期流程：**

1. 技能读取你现有的 SKILL.md，映射其职责
2. 根据上述信号识别自然分割点
3. 提出拆分方案：哪些部分成为技能，哪些成为共享资源
4. 你批准后，转入标准三阶段访谈，并携带更丰富的上下文

**常见模式：** 分支拆分（if/else 分支 → 独立技能）、流水线提取（顺序阶段 → 链式技能）、参考资料提取（内联表格 → `references/` 文件）。并非每个大型技能都需要拆分 — 如果技能虽然很长但有单一明确的职责，提取到 `references/` 即可。

### 场景 C：组合（合并多个技能）

**你有多个独立技能，想让它们作为协调的 bundle-plugin 协作。**

**预期流程：**

1. 技能清点所有候选技能（来源、结构、质量、许可证）
2. 检查兼容性（命名冲突、职责重叠、风格一致性）
3. 对第三方技能，询问集成意图：
   - **原样打包** — 不修改直接捆绑（添加来源标注）
   - **集成到工作流** — 适配描述、交叉引用和交接
4. 设计编排（工作流链路、粘合技能、共享资源、引导）
5. 你批准后，转入标准三阶段访谈，并携带更丰富的上下文

**第三方技能处理：**

| 步骤 | 必需？ | 目的 |
|------|:---:|------|
| 许可证检查 | 是 | 确保许可证兼容性 |
| 安全审计 | 是 | 对导入内容运行 `bundles-forge:auditing` |
| 来源标注 | 是 | 在复制的 SKILL.md 中添加来源头 |
| 描述重写 | 仅工作流集成 | 将触发条件适配到新上下文 |
| 交叉引用重写 | 仅工作流集成 | 将 `old-project:skill` 改为 `new-project:skill` |

**绝不自动安装存在未解决严重安全发现的第三方技能。**

## 设计文档

所有场景都产出如下结构的设计文档：

```markdown
## Bundle-Plugin Design: <project-name>

### Project Overview
<一句话说清楚：这个技能包是什么、解决什么问题>

### Target Users
<谁会用、什么背景、典型工作流>

### Use Cases
<2-3 个具体场景>

### Success Criteria
<怎么判断这个技能包做得好不好>

**Mode:** quick / adaptive
**Platforms:** <list>
**Bootstrap:** yes/no

### Skills
| Name | Purpose | Type | Visibility | Dependencies |
|------|---------|------|------------|--------------|

### Workflow Chain
<description or "independent">

### Advanced Components
<list or "none">

### Third-Party Sources (if applicable)
| Skill | Source | License | Integration Intent |
|-------|--------|---------|-------------------|

### Notes
<special requirements or constraints>
```

**批准前检查清单：**
- [ ] 项目概述清晰地表述了要解决的问题
- [ ] 目标用户具体（不只是"开发者"）
- [ ] 所有技能名使用 kebab-case
- [ ] 工作流依赖已映射（或标记为独立）
- [ ] 平台选择与实际使用一致
- [ ] 第三方技能已注明许可证和安全审计情况
- [ ] 成功标准尽可能可量化

## 流水线：批准后的自动流程

设计文档获批后，**蓝图规划会编排**四步流水线。这不是仅向脚手架做一次自动交接；编排器按顺序推进各步骤：

| 步骤 | 执行内容 | 目的 |
|------|----------|------|
| **Scaffold** | `bundles-forge:scaffolding` | 生成项目结构与各平台清单 |
| **Author Content** | `bundles-forge:authoring` | 撰写 **SKILL.md** 与 **agents/*.md** — 由**蓝图规划**在脚手架之后调用（**不由** scaffolding 调用） |
| **Wire Workflow** | 工作流设计（蓝图规划内部执行） | 按设计将技能串联（链路、引导、命令等） |
| **Run Audit** | `bundles-forge:auditing` | 在进一步迭代前做质量与安全校验 |

每个阶段都包含验证门禁：agent 会检查自身的产出（如 inspector 通过、frontmatter 有效、工作流链路对称、无严重审计发现），发现问题时会回退修复后再继续。如果最终审计出现警告，agent 会呈现警告并询问是否继续。

若后续需要增加平台，可再次使用 `bundles-forge:scaffolding`。

Agent 会携带你已批准的设计文档贯穿上述阶段。**authoring** 作为该流水线的一环由**蓝图规划**调用；脚手架不会调用 authoring。设计文档的需求上下文（项目概述、目标用户、使用场景）会传递给 authoring，帮助写出更有针对性的技能描述。

## 常见问题

**Q：Agent 跳过访谈直接进入了脚手架，怎么回事？**

技能有 HARD-GATE 防止这种情况。如果发生了，说明技能未加载 — 显式调用 `/bundles-blueprint` 以确保完整访谈运行。Agent 必须在你批准设计文档后才能进入脚手架。

**Q：Agent 不停追问不往前推进，怎么加速？**

每个阶段都有检查点 — 信息验证通过后 agent 才会推进。如果想走更短的流程，早点告诉 agent 你在做"快速打包"。快速模式只问 2 个需求问题 + 4 个架构问题。你也可以说"细节够了，继续推进"，agent 会评估是否有足够信息生成设计文档。

**Q：Agent 给了多个方案，我不知道选哪个。**

仔细看 agent 提供的优劣分析。Agent 总会标注一个"推荐"并说明理由。如果不确定，就选推荐的 — 之后可以调整。如果都不合适，告诉 agent 哪里不对，它会提出替代方案。

**Q：访谈问的问题太多，我的项目很简单。**

你可能处于自适应模式。在架构阶段，agent 本应为简单项目推荐快速模式。如果没有，直接告诉它你要快速模式。

**Q：设计文档有问题，能回去改吗？**

可以。在审核门禁处，你可以要求修改设计的任何部分。Agent 会更新文档并重新自审。你也可以要求回到某个具体阶段重新讨论决策，无需从头开始。

**Q：第三方技能安全审计失败了。**

修复导入技能中的安全问题，或将其从组合中排除。绝不批准包含存在未解决严重安全发现的第三方技能的设计。

**Q：不确定该选哪个场景。**

参照"选择场景"章节的决策流程图。如果不确定，从场景 A（新项目）开始 — 如果实际需要拆分或组合，agent 可以适配调整。

**Q：我想向现有项目添加技能，而不是创建新项目。**

使用 `/bundles-optimize` — 目标 7（技能与工作流重构）处理在现有项目中添加、替换和重组技能。蓝图规划仅用于创建新项目。

**Q：蓝图规划和脚手架有什么区别？**

蓝图规划是*规划*阶段 — 它访谈你并产出设计文档。脚手架是*执行*阶段 — 它读取设计并生成实际的项目文件。对于在现有项目上的平台适配，你可以跳过蓝图规划，直接调用脚手架。

**Q：我可以跳过蓝图规划直接 scaffold 吗？**

可以，但仅限于平台适配（在现有项目上添加/移除平台支持）。对于新项目，跳过访谈意味着 agent 需要猜测你的需求 — 这会导致返工。5 分钟的蓝图规划几乎总是值得的。
